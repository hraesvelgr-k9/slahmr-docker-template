A Docker + Docker Compose template for working with **SLAHMR** in both development and production-like workflows.

This template uses a two-mode setup:

- **Development mode**: bind-mount the host-side `workspace/slahmr` source tree so code changes are reflected immediately inside the container. ViTPose and DROID-SLAM are installed in editable/build mode at first startup and re-verified on every container launch.
- **Production-like mode**: copy the source tree into the image to improve reproducibility and reduce dependency on host-side mounts.

## Features

- Development workflow with host-side source editing via bind-mount
- Production-like workflow with image-baked source code
- `.env`-based configuration for container name, GPU architecture, paths, and image settings
- `setup.sh` for one-time host-side preparation (clone, entrypoint copy, SMPL + weight download, placeholder config generation)
- `Makefile` shortcuts for all Docker Compose tasks, preprocessing, and inference
- Import-verified bootstrap: `entrypoint.dev.sh` re-runs ViTPose / DROID-SLAM setup if `mmpose` or `droid_backends` cannot be imported, even when a state stamp already exists
- `clean-pycache` target using a temporary Alpine container to remove root-owned `__pycache__` directories
- Hydra config bind-mounts for `custom.yaml` and `video.yaml` (single-file mounts into the container's `slahmr/confs/data/`)
- `.dockerignore` to reduce unnecessary build context

## Quick Start

### 1. Create your environment file

```bash
cp .env.example .env
```

### 2. Clone SLAHMR and run host-side setup

```bash
make init
make setup
```

`make init` clones the SLAHMR source into `workspace/slahmr`.  
`make setup` copies entrypoint scripts, downloads the SMPL model and pre-trained weights, and creates placeholder Hydra config files under `configs/`.

### 3. Build and start development mode

```bash
make dev-build
make dev-up
make dev-shell
```

On first startup, `entrypoint.dev.sh` installs ViTPose (editable) and builds DROID-SLAM automatically.

## Common Commands

### Setup

```bash
make env-init        # Copy .env.example -> .env (if absent)
make init            # Clone SLAHMR source into workspace/slahmr
make setup           # Run setup.sh (entrypoints + weights + configs)
make reinit          # Remove workspace/slahmr and re-clone from scratch
make clean-pycache   # Remove root-owned __pycache__ via a temporary container
```

### Development mode

```bash
make dev-build       # Build dev image
make dev-up          # Start dev container (detached)
make dev-down        # Stop dev container
make dev-shell       # Open bash in running dev container
make dev-logs        # Tail dev container logs
make dev-config      # Show merged dev Compose config
make dev-rebuild     # Rebuild and restart dev container
make dev-restart     # Reset bootstrap stamp and restart container
```

### Production-like mode

```bash
make prod-build      # Build prod image
make prod-up         # Start prod container (detached)
make prod-shell      # Open bash in running prod container
make prod-logs       # Tail prod container logs
make prod-config     # Show merged prod Compose config
make prod-rebuild    # Rebuild and restart prod container
make prod-down       # Stop prod container
```

### Preprocessing & Inference

```bash
make prepare-custom VIDEO=/workspace/data/inputs/sample.mp4 [SEQ=my_seq]
make prepare-video  VIDEO=/workspace/data/inputs/sample.mp4 [SEQ=my_seq] [FPS=30]
make run-custom
make run-video
```

### Utilities

```bash
make ps              # Show container status
make clean           # Remove containers and named volumes
```

## Configuration

Main settings are defined in `.env.example`. Copy it to `.env` and adjust as needed.

| Variable | Description |
|---|---|
| `IMAGE_NAME` / `IMAGE_TAG` | Docker image name and tag |
| `CONTAINER_NAME` | Name assigned to the running container |
| `CONTAINER_SHM_SIZE` | Shared memory size (default: `16gb`) |
| `TORCH_CUDA_ARCH_LIST` | CUDA architectures to compile for |
| `HOST_SLAHMR_DIR` | Host path to the SLAHMR source tree |
| `HOST_SCRIPTS_DIR` | Host path to the scripts directory |
| `HOST_CUSTOM_CFG_FILE` | Host path to `configs/custom.yaml` |
| `HOST_VIDEO_CFG_FILE` | Host path to `configs/video.yaml` |
| `HOST_DATA_DIR` | Host path to the data directory |
| `CONTAINER_SLAHMR_DIR` | Container path for the SLAHMR source |
| `CONTAINER_CUSTOM_CFG_FILE` | Container path for `custom.yaml` inside Hydra confs |
| `CONTAINER_VIDEO_CFG_FILE` | Container path for `video.yaml` inside Hydra confs |
| `CONTAINER_DATA_DIR` | Container path for the data directory |
| `DATA_SPLIT` / `RUN_OPT` / `RUN_VIS` | Runtime defaults for inference |

## Using Multiple Compose Files

This template uses a base Compose file plus mode-specific override files:

- `compose.yml` вЂ” shared service definitions, named volumes, GPU runtime
- `compose.dev.yml` вЂ” development overrides (bind-mounts, dev image target, dev entrypoint)
- `compose.prod.yml` вЂ” production-like overrides (prod image target, prod entrypoint)

Docker Compose merges files in the order they are specified. Later files override or extend earlier ones.

Examples:

```bash
docker compose --env-file .env -f compose.yml -f compose.dev.yml up -d --build
docker compose --env-file .env -f compose.yml -f compose.prod.yml up -d --build
```

## Development vs Production-like

### Development mode

Use development mode when:

- You want to edit SLAHMR source code on the host machine
- You want source changes to appear immediately inside the container
- You want faster iteration with fewer rebuilds

### Production-like mode

Use production-like mode when:

- You want to verify reproducibility for a specific source state
- You want to run a container without depending on a host bind-mount
- You want behaviour closer to a distributable image

## Preprocessing & Inference

Place input video files under `data/inputs/` on the host (bind-mounted to `/workspace/data/inputs/` inside the container).

### Custom sequence pipeline

```bash
# 1. Preprocess (runs inside container via docker compose run --rm)
make prepare-custom VIDEO=/workspace/data/inputs/sample.mp4 SEQ=my_seq

# 2. Run SLAHMR inference
make run-custom
```

`VIDEO` is required. `SEQ` defaults to the basename of the video file if omitted.

Under the hood, `make prepare-custom` invokes `scripts/prepare_custom.sh` inside the container, and `make run-custom` invokes `scripts/run_custom_demo.sh`.

### Video pipeline

```bash
# 1. Preprocess with optional FPS resampling
make prepare-video VIDEO=/workspace/data/inputs/sample.mp4 SEQ=my_seq FPS=30

# 2. Run SLAHMR inference
make run-video
```

`make prepare-video` invokes `scripts/prepare_video.sh` and `make run-video` invokes `scripts/run_video_demo.sh` inside the container.

Both `prepare-video` and `run-video` use the `configs/video.yaml` Hydra config, which is bind-mounted into the container as `slahmr/confs/data/video.yaml` at startup.

### Scripts reference

| Script | Invoked by | Purpose |
|---|---|---|
| `scripts/prepare_custom.sh` | `make prepare-custom` | Detect people, run HMR2, build a custom sequence |
| `scripts/prepare_video.sh` | `make prepare-video` | Same as above with optional FPS resampling |
| `scripts/run_custom_demo.sh` | `make run-custom` | Run SLAHMR optimisation on a custom sequence |
| `scripts/run_video_demo.sh` | `make run-video` | Run SLAHMR optimisation on a video sequence |

All four scripts live under `scripts/` on the host and are bind-mounted into the container via `HOST_SCRIPTS_DIR` / `CONTAINER_SLAHMR_DIR` settings in `.env`.

## Bootstrap Behaviour

`entrypoint.dev.sh` performs an **import-based bootstrap check** on every container launch:

1. If `configs/video.yaml` or `configs/custom.yaml` does not yet exist, run `make setup` first to generate them.
2. On container start, `preflight_hmr2()` ensures `hmr2_data` is extracted into `/root/.cache/4DHumans` before any Python code runs. If a pre-downloaded tarball exists at `/workspace/data/models/hmr2_data.tar.gz`, it is extracted locally; otherwise the remote download path is used.
3. After the model cache step, `need_bootstrap()` checks whether `mmpose` and `droid_backends` can actually be imported. If either check fails вЂ” even when the state stamp `/var/lib/slahmr/.deps_installed` already exists вЂ” `bootstrap_dev()` reinstalls ViTPose (editable) and rebuilds DROID-SLAM.

This means `docker compose run --rm` in a fresh container against an existing named volume will automatically recover a broken environment without manual intervention.

To force a full re-bootstrap manually:

```bash
make dev-restart
```

## Troubleshooting

### 1. My changes are not visible in the container

Check that `HOST_SLAHMR_DIR` in `.env` points to the correct host directory and that the service is running in **development mode**. In development mode, the bind-mounted directory takes precedence over any image-baked files.

### 2. ViTPose or DROID-SLAM is not importable after restart

`entrypoint.dev.sh` will detect the import failure and re-run the bootstrap automatically. If it fails repeatedly, check that `workspace/slahmr/third-party/ViTPose` and `workspace/slahmr/third-party/DROID-SLAM` exist on the host (populated by `make init`).

### 3. `__pycache__` directories cannot be deleted on the host

Files created inside the container are owned by root. Use:

```bash
make clean-pycache
```

This launches a temporary Alpine container to remove them as root.

### 4. `${VAR}` is not expanding in Compose

Compose interpolation reads from `.env` or `--env-file`. Verify that `.env` exists in the project root, or pass `--env-file .env` explicitly.

### 5. Compose bind-mount fails at startup with "file not found"

`configs/custom.yaml` and `configs/video.yaml` must exist before starting the container because they are single-file bind-mounts. Run `make setup` to generate them as placeholders.

### 6. Development mode works but production-like mode fails

Ensure `workspace/slahmr` is present in the build context and not excluded by `.dockerignore`. The production-like image depends on `COPY workspace/slahmr /workspace/slahmr` during build.

### 7. GPU is not available inside the container

Check the host NVIDIA driver and Docker GPU runtime configuration. Confirm the container is started with the GPU-related environment variables from `.env`.

### 8. I want to inspect the final merged Compose configuration

```bash
make dev-config
make prod-config
```

This helps verify merged settings, interpolated values, and resolved paths before debugging runtime behaviour.

## Recommended Repository Practice

For public or team distribution:

- Commit `.env.example`, not `.env`
- Commit `.gitignore` and `.dockerignore`
- Keep large datasets, checkpoints, outputs, and model files out of Git
- Keep `workspace/slahmr` as a user-provided clone target (`make init` handles this)
- Commit `configs/custom.yaml` and `configs/video.yaml` as placeholder files only; their actual content is overwritten at runtime by the preprocessing scripts
- Commit all four scripts under `scripts/` вЂ” they are the host-side entry points for preprocessing and inference
- Document expected GPU architecture values in `.env.example`
